# 文档：Sidecar 架构与 Monkey Patch 实践说明

## 1. 方案背景 (Context)

在基于开源框架（如 `nanobot`）进行深度定制时，传统的 `Fork -> Modify -> Merge` 路径往往会带来沉重的维护成本。为了在不破坏上游代码纯净度的前提下实现个性化扩展，本项目采用了 **Sidecar 架构** 结合 **Monkey Patch (猴子补丁)** 的技术方案。

---

## 2. 核心架构：Sidecar 模式

本项目建立了一个独立的 `cafeext/` 目录，作为系统的 Sidecar。

### 2.1 零代码冲突
原始的 `nanobot` 源码目录保持 100% 纯净。所有定制逻辑、私有配置、加密 Key 甚至 Prompt 模板都存放在外挂目录中。无论上游（Upstream）更新了多少个 PR，只需执行 `git pull` 即可，极大降低了合并冲突的概率。

### 2.2 私有化隔离
`cafeext/` 目录可作为独立的代码库进行管理，甚至可以存放在私有的代码仓库中。这确保了在参与开源贡献或公开展示代码时，私有的配置和凭据（如 `config.json` 或 `.env`）不会被泄露。

---

## 3. 实现手段：Monkey Patching

利用 Python 的动态特性，我们在系统启动阶段（`launcher.py`）对 `nanobot` 的核心类和方法进行运行时修改。

### 3.1 安全拦截
拦截 `ToolRegistry.execute` 方法，在不修改源码的情况下，强制为敏感工具（如 `exec`）注入 macOS 权限确认对话框。

### 3.2 记忆与提示词补强
- **双层记忆**: 拦截 `ContextBuilder`，动态合并金库（Vault）中的核心设定与工作区（Workspace）中的动态事实。
- **提示词替换**: 直接在内存中定位并替换硬编码的提示词片段，实现全中文环境适配。

---

## 4. 维护与验证

虽然这种方案避开了源码层面的冲突，但其稳定性依赖于拦截点的持久有效性。

### 4.1 逻辑验证
当上游核心代码发生重大重构（如方法签名变更）时，补丁可能失效。因此，在更新 `nanobot` 后，必须通过以下方式验证：
- 运行诊断工具：使用 `nb doctor` 检查环境完整性。
- 逻辑回归：确保关键的注入逻辑（如安全确认）依然正常工作。

### 4.2 最小化拦截准则
为了降低耦合度，补丁逻辑应尽量只作用于系统的「入口」和「出口」位置（如 CLI 命令层、工具执行层、消息总线层），避免深入复杂的中间业务逻辑。

总结                                                                                                                     
如果你想长期跟踪上游更新，又想做深度的个性化（比如加个猫娘人格、改个存储路径、加个安全沙箱），Sidecar + Monkey Patch
是目前平衡「开发爽感」与「维护成本」的最优解。
