# Research: 评估 upstream main v0.1.5 对 sidecar 的合并影响

## 1. Research Question
- **核心问题**: 上游 `main` 的 `v0.1.5` 更新相对当前 sidecar 分支引入了哪些功能热区，哪些能力与 `ava/` 已有实现发生冲突、重叠或可以收窄/删除。
- **调研范围**:
  - 拉取并比对 `upstream/main` 上与 `v0.1.5` 相关的代码/文档变更
  - 盘点命中的 `nanobot/` 热区与 `ava/patches/*`、`ava/forks/*`、`ava/tools/*` 的重叠面
  - 输出合并策略建议：保留 / 收窄 / 删除 / 上推 / 暂缓
- **范围边界（不调研什么）**:
  - 不直接执行代码合并
  - 不在本任务中实现 patch 重构或回归修复
  - 不展开与 `v0.1.5` 无直接关联的 sidecar 长期重构
- **成功标准（什么算调研完成）**:
  - 给出 `v0.1.5` 相对当前 sidecar 的主要变更面
  - 明确列出至少一版热区冲突/重复实现清单
  - 为后续真正 merge 提供可执行的优先级和验证建议
- **决策背景**:
  - 当前仓库以 `ava/` sidecar + monkey patch 方式扩展 `nanobot`
  - 该仓库此前已经做过多轮 `upstream/main` merge，已有 `.specanchor/patch_map.md` 和 merge follow-up spec 可复用
  - 本次任务目标是先做 repo-grounded 分析 spec，避免直接 merge 后才发现热区冲突

## 2. Explore
### 2.1 调研方法
- `git fetch upstream --tags` 获取最新上游 tags / commits
- 用 `git diff` / `git log` / `git merge-base` 盘点 `v0.1.5` 相对当前基线的真实代码触点
- 对照 `.specanchor/patch_map.md`、module specs 和 `ava/` 代码，识别冲突/重复实现
- 额外区分三种关系：
  - 文本级 merge 冲突：同一上游文件在当前分支和 `v0.1.5` 都被改动
  - 行为级热区重叠：上游改动命中 `ava/patches/*` / `ava/forks/*` 拦截面
  - 同域但不重复：功能都在同一能力域，但职责层次不同，不应误判为“可直接删”

### 2.2 调研过程
#### 方向 1: 上游版本面扫描
- 调研内容: 确认 `v0.1.5` tag、当前 sidecar 已合入的 upstream 基线、以及 `upstream/main` 与 tag 的关系
- 关键发现:
  - 当前 sidecar 记录的 upstream 基线是 `7113ad34f4e8259fea0483b5a1835ccc66797610`，即 `ava/UPSTREAM_VERSION` 中记录的 2026-04-03 已验证基线
  - `v0.1.5` tag 对应 `79234d237e840b144c73926a0cc1df6e810335d8`，提交时间为 2026-04-06 11:26:07 UTC
  - 当前 `upstream/main` 是 `e21ba5f6675a2aa20d1f07c897b5d3e6775cc29c`，提交时间为 2026-04-07 17:10:27 UTC
  - `git merge-base v0.1.5 upstream/main = v0.1.5`，且 `git rev-list --left-right --count v0.1.5...upstream/main = 0 48`
- 数据/证据:
  - `git show --no-patch v0.1.5`
  - `git show --no-patch upstream/main`
  - `git merge-base v0.1.5 upstream/main`
  - `git rev-list --left-right --count v0.1.5...upstream/main`

#### 方向 2: sidecar 热区映射
- 调研内容: 将 `7113ad34..v0.1.5` 命中的上游文件映射到 patch/fork/tool/spec 热区，并单独拆出 `04a41e31..v0.1.5` 的新增波次
- 关键发现:

  - 第二波（`04a41e31..v0.1.5`）主要新增 `provider retry/structured error`、`Qianfan/Langfuse`、`voice transcription`、`email attachments`、`sandbox hardening`、`config env var interpolation`
- 数据/证据:

  - `git diff --dirstat=files,0 7113ad34..v0.1.5 -- nanobot`
  - `git log --oneline --reverse 7113ad34..v0.1.5`
  - `git diff --name-only 04a41e31..v0.1.5 -- nanobot`

#### 方向 3: 重复实现与收敛机会
- 调研内容: 判断哪些 sidecar 能力已被上游吸收，哪些仍需保留，并识别真实文本冲突文件
- 关键发现:
  - 当前分支与 `v0.1.5` 之间，真正的共享上游文件冲突候选只有 3 个：`nanobot/agent/context.py`、`nanobot/config/loader.py`、`nanobot/templates/TOOLS.md`
  - 但行为级热区不止这 3 个；`schema/onboard`、`loop/context`、`tools/templates`、`console/restart`、`skills` 仍是 merge 后必须复核的 patch 面
  - “重复实现”最明显的不是大块业务功能，而是控制面和提示词面：`context` 清洗、`TOOLS.md` 工具说明、restart 状态/通知
- 数据/证据:
  - `comm -12 <(git diff --name-only 7113ad34..v0.1.5 -- nanobot | sort) <(git diff --name-only 7113ad34..HEAD -- nanobot | sort)`
  - `sed -n` 读取 `ava/patches/*.py`、`ava/forks/config/schema.py`、`ava/templates/TOOLS.md`

- 本次未做运行时实验；目标是先产出 merge 研究结论，避免在未定范围时进入实现
- 已做的“静态实验”只有版本拓扑、文件触点交集、patch 热区映射

## 3. Findings
### 3.1 关键事实
1. 当前 sidecar 并不是从 `v0.1.5` 起步，而是停在更早的 upstream 基线 `7113ad34`；因此本轮分析的真实范围是 `7113ad34..v0.1.5`，而不是“只看 tag 当天那一个 bump commit”。
2. `v0.1.5` 不是当前 `upstream/main` 的终点。到 2026-04-08，本地抓到的 `upstream/main` 已经比 `v0.1.5` 再前进 48 个提交，所以后续必须明确是“先并 tag”还是“直接并 main”。
3. `7113ad34..v0.1.5` 影响面很大，但真正与当前分支形成文本级共享冲突的上游文件只有 3 个：`nanobot/agent/context.py`、`nanobot/config/loader.py`、`nanobot/templates/TOOLS.md`。这意味着代码 merge 冲突面并不大，真正复杂的是行为级重叠。
4. 第一波高风险热区仍然是 `patch_map.md` 已经预警的几块：`schema/onboard`、`console/restart`、`context/loop`、`tools/templates`。第二波到 `v0.1.5` 则主要增加 provider/channel/config 周边能力，和 sidecar 是“同域但不完全重复”的关系。
5. 真正值得警惕的“重复实现”主要有三类：
   - `context.py` 内的 `_sanitize_history()` 与 `ava/patches/context_patch.py` 的 provider 侧 `sanitize_messages()` 出现双保险重叠
   - `nanobot/templates/TOOLS.md` 的本地重写与 `ava/templates/TOOLS.md` 的 overlay 形成双事实源，上游新增 `glob/grep` 文档不会自动体现在运行时有效模板里
   - upstream `utils/restart.py` 的 CLI 重启通知与 sidecar `LifecycleManager + gateway_control + console_patch` 都在做“重启状态表达”，如果不收口会出现双来源状态

### 3.2 对比分析（如有多方案）
| 热区 | upstream `v0.1.5` 变化 | 当前 sidecar 面 | 关系判定 | 合并建议 |
|------|----------------------|------------------|----------|----------|
| `context` 提示词与历史清洗 | `render_template()` 模板化 identity/skills；`build_messages()` 合并同角色消息 | `nanobot/agent/context.py` 本地 `_sanitize_history()`；`ava/patches/context_patch.py` 做摘要/压缩/记忆注入与 provider 侧 sanitize | **直接冲突 + 局部重复实现** | 以 upstream 模板化 prompt 为主，保留 `ava/context_patch` 作为主清洗层，再判断本地 `_sanitize_history()` 是否可收窄或移除 |
| `config/schema/onboard` | 新增 `DreamConfig`、`WebToolsConfig.enable`、`ssrf_whitelist`、`xiaomi_mimo`；`loader.py` 支持 `${VAR}` env 插值 | `a_schema_patch` 继承式 fork；`c_onboard_patch` 保形刷新；本地 `nanobot/config/loader.py` 支持 `extra_config.json` overlay | **loader 直接冲突 + schema/onboard 行为热区** | 合并时把 `env var` 与 `extra_config overlay` 组合进同一 loader 路径，并专门验证 `Dream` / `web.enable` / `ssrf_whitelist` 的落盘与 refresh |
| `tools` 原生能力与工具说明 | 原生新增 `GlobTool` / `GrepTool`、`Schema` / `tool_parameters`、`nanobot/templates/TOOLS.md` 补文档 | `tools_patch` 追加 sidecar 工具；`templates_patch` 以 `ava/templates/TOOLS.md` 覆盖运行时模板 | **代码面低冲突，文档面高漂移** | 工具注册先不删 sidecar；优先同步 `ava/templates/TOOLS.md`，否则 merge 后 runtime 仍看不到 `glob/grep` 指引 |
| `console/restart` 控制面 | `utils/restart.py`、CLI restart notice、gateway 启动路径继续演进 | `console_patch`、`LifecycleManager`、`gateway_control`、console `/restart` API | **同域但层次不同** | 不要把 upstream restart notice 当第二控制面；建议把它视为本地 lifecycle 的一个信号源或兼容桥 |
| `skills` 加载器 | `SkillsLoader` 内部重构、frontmatter 处理与 summary 生成改写 | `skills_patch` 重写 `__init__/list_skills/load_skill`，提供 `ava/.agents/nanobot` 三源发现和 disabled filter | **方法级热区重叠** | merge 后必须回归 `skills_patch`；短期不删，长期可评估是否能收窄到只补 sidecar 特有策略 |
| `memory/storage` | `Dream`、`Consolidator`、`GitStore`、历史注入细节增强 | `categorized_memory`、`history_summarizer`、`history_compressor`、SQLite `storage_patch`、`MemoryTool` | **同域但职责不同** | 先保留 sidecar 记忆栈；重点复核 `/new`、历史注入、后台任务 digest 是否和 upstream Dream 相互干扰 |
| `transcription/provider` | Whisper/Groq 转写统一、provider retry 分类、`Qianfan/Langfuse/GPT-5` | `transcription_patch` 只补 Groq 代理；`codex` / console token 面依赖 provider 输出 | **大多是上游增益** | 作为低优先级回归区处理，重点确认代理 patch 没挡住上游统一转写链路 |

### 3.3 Trade-offs
- **只对齐 `v0.1.5` tag 再说**:
  - Pros: 范围比 `upstream/main` 小，能先把 `Dream/web_config/glob-grep` 这一波吃下来
  - Cons: 这只是追到 2026-04-06 的 tag，距离当前 `upstream/main` 还差 48 个提交，可能很快又要再做一轮 merge 研究
- **直接瞄准 `upstream/main`**:
  - Pros: 一次性追到最新主线，避免先并 tag 再立刻补并 main
  - Cons: 额外 48 个提交会把 `memory / mcp / tool hints / cron / telegram geo` 等问题一起带进来，分析面和验证面明显扩大
- **先收口当前分支的直接 `nanobot/` 定制，再做 merge**:
  - Pros: 能把真实 merge 冲突从“共享文件 + patch 热区”缩回“纯 upstream + ava patch”
  - Cons: 这本身就是一个前置重构任务，不适合和上游合并绑成同一回合

### 3.4 未解决的问题
- 后续真正执行合并时，目标应该是 `v0.1.5` 还是 `upstream/main@HEAD`？
- 当前直接落在 `nanobot/config/loader.py` 和 `nanobot/templates/TOOLS.md` 的 sidecar 行为，是否要先迁回 `ava/` 侧，降低未来 merge 成本？
- `context` 清洗应该以 upstream `build_messages()` 为主，还是以 `ava/context_patch` provider 侧 sanitize 为主？若保留双层清洗，需要明确各自边界
- `restart` 最终是否只保留 `LifecycleManager` 为状态真源，把 upstream `restart_notice` 降为兼容提示？

## 4. Challenge & Follow-up
### 4.1 Agent 追问
- 如果下一步本来就准备投入一轮完整 merge + 回归，那是否还值得把执行目标停在 `v0.1.5`，而不是直接对齐到 `upstream/main`？
- 当前真正会反复制造 merge 成本的，不是 `ava/patches/*`，而是仍直接落在 `nanobot/` 的 sidecar 定制点；要不要把它们视为前置收口对象，而不是每次都手工 reconcile？
- `context` / `restart` / `TOOLS.md` 这些控制面能力里，哪些应该以 upstream 为真源，哪些应该继续由 sidecar 持有真源？如果这个边界不先定，代码层 merge 会反复拉扯

### 4.2 用户反馈

### 4.3 方向调整（基于追问）
- 本轮不再把目标局限于“解释 v0.1.5 改了什么”，而是把 `v0.1.5` 当成一个有边界的热区地图
- 结论以“下一步怎么 merge 更省总成本”为主，而不是以 tag 本身为终点
- 对于 `context`、`loader`、`TOOLS.md` 这 3 个共享文件，结论直接按“先定真源，再做 reconcile”组织

## 5. Conclusion
### 5.1 Action Items
- [ ] 1. 先确定执行目标：
  - 若目标是“尽快追到最新 upstream，减少后续重复 merge 成本”，直接以 `upstream/main` 为目标
  - 若目标是“先做一个 release-aligned、范围可控的 checkpoint”，先并 `v0.1.5`
- [ ] 2. 在真正 merge 前，先为 3 个共享冲突文件准备手工 reconcile 策略：
  - `nanobot/agent/context.py`：以上游模板化 prompt 为主；sidecar 继续保留 `context_patch` 的摘要/压缩/记忆注入；评估是否删除或收窄本地 `_sanitize_history()`
  - `nanobot/config/loader.py`：把 upstream `${VAR}` env 插值与本地 `extra_config.json` overlay 合并到一条加载链；确保 `c_onboard_patch` 的“保形刷新”不被破坏
  - `nanobot/templates/TOOLS.md` / `ava/templates/TOOLS.md`：同步 `glob` / `grep` 等上游新增工具说明到 sidecar overlay，避免 merge 后运行时模板继续陈旧

### 5.2 最终建议
- **推荐方案**: 如果下一步就是要实际合并，**不建议把执行目标停在 `v0.1.5`**；更划算的做法是把本 spec 当作 `v0.1.5` 热区地图，然后直接对齐 `upstream/main`，除非你明确需要一个 release-aligned checkpoint
- **推荐理由**:
  - `v0.1.5` 已经落后当前 `upstream/main` 48 个提交，先并 tag 再并 main 很可能重复做两轮相同的控制面 reconcile
  - `v0.1.5` 暴露出的核心冲突并不是“这个 tag 独有的问题”，而是当前 sidecar 与 upstream 在 `context` / `loader` / `TOOLS.md` / `restart` / `skills` 这些长期热区上的边界问题
  - 如果这些边界不先收口，后续每轮 upstream sync 都会反复撞同一批点
- **风险提示**:
  - 直接对齐 `upstream/main` 会把 `v0.1.5` 后的 48 个提交一起带进来，尤其是 `memory`、`mcp`、`tool hints`、`cron`、`telegram geo` 这几块会扩大验证面
  - 当前仍有 sidecar 行为直接落在 `nanobot/config/loader.py` 与 `nanobot/templates/TOOLS.md`，这是未来 merge 成本持续偏高的结构性原因
  - `restart` 目前有 upstream notice、sidecar lifecycle、console gateway 三套表达层；如果不先定义真源，merge 后最容易出现“状态看起来都对，但不是同一语义”的假闭环
- **下一步**:
  - 若只是要保守推进：新建一个实施型 Task Spec，目标定为“merge `v0.1.5` + 收口 3 个共享文件”
  - 若要一次性追平主线：新建一个实施型 Task Spec，目标定为“merge `upstream/main`，以本 spec 的热区顺序分批执行”
